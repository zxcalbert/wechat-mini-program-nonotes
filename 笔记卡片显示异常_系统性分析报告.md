## 二、已执行的修复

### 2.1 formatLetters 过滤逻辑修复

**问题代码**（原第251行）：
```javascript
.filter(item => !item.deleted)  // ❌ 错误
```

**修复代码**（现第251行）：
```javascript
.filter(item => item.deleted !== true)  // ✅ 正确
```

**问题分析**：
```javascript
// 原逻辑问题：
!item.deleted  → 当 item.deleted = false 时，!false = true → 笔记被过滤！ ❌
!item.deleted  → 当 item.deleted = undefined 时，!undefined = true → 笔记保留 ✅

// 修复后逻辑：
item.deleted !== true  → 当 item.deleted = false 时，false !== true = true → 保留 ✅
item.deleted !== true  → 当 item.deleted = undefined 时，undefined !== true = true → 保留 ✅
item.deleted !== true  → 当 item.deleted = true 时，true !== true = false → 过滤 ✅
```

## 四、问题根源推测

### 4.1 主要推测：数据库 deleted 字段数据不一致

| 推测 | 说明 |
|------|------|
| **数据库中部分笔记的 `deleted` 字段为 `false`** | 在原逻辑 `.filter(item => !item.deleted)` 下，这些笔记被错误过滤 |
| **二次编译时使用了缓存数据** | 缓存中的数据格式或内容与数据库不同 |

### 4.2 数据流程差异

**首次登录（清除缓存后）**：
```
登录获取 openid
    ↓
数据库查询 (10条)
    ↓
其中 6 条 deleted=false, 4 条 deleted=undefined
    ↓
原逻辑：.filter(item => !item.deleted)
    ↓
!false = true → 6条被过滤
!undefined = true → 4条保留
    ↓
只显示4条 ← 这就是问题！
```

**二次编译（有缓存）**：
```
openid 已缓存
    ↓
从缓存读取 (10条)
    ↓
缓存中的数据格式或 deleted 字段不同
    ↓
正常显示10条
    ↓
滑动加载更多 ← 恢复正常
```
