## 三、🎯 根本原因确认

### 3.1 问题本质

**不是代码逻辑错误，而是数据库查询策略问题！**

### 3.2 数据库实际情况

```
letters 集合中的笔记（按 createTime 倒序排列）：

第1页 (skip=0, limit=10) 的10条记录：
├── 5条 deleted = undefined (未删除) → 应该显示 ✅
├── 5条 deleted = true     (已删除) → 应该过滤 ❌
└── 过滤后只剩：5条 ❗❗❗

第2页 (skip=10, limit=10) 的10条记录：
├── 10条全部 deleted = undefined (未删除) → 全部显示 ✅
└── 过滤后保留：10条 ✅✅✅

总计有效笔记：5 + 10 = 15条+
```

### 3.3 为什么会出现这种数据分布？

可能的原因：
1. **批量删除操作**：之前某次批量删除了较新的5条笔记（标记为 deleted=true）
2. **时间顺序**：被删除的笔记恰好都是最近创建的（createTime 较大），排在前面
3. **数据迁移**：从旧系统迁移数据时，部分旧数据的 deleted 字段未正确设置

### 3.4 为什么二次编译能看到更多？

**因为用户手动滑动到了底部，触发了 `loadMoreLetters()` 加载了第2页！**

而首次登录时：
- 页面显示5条（第1页过滤后的结果）
- 如果页面高度足够显示5条，用户不会滚动
- 所以不会触发加载更多

---

## 四、解决方案

### 4.1 核心思路

**在数据库查询时就排除已删除的笔记，而不是获取后再过滤！**

这样可以确保每页都返回10条有效的未删除笔记。

### 4.2 实施方案

**文件**：`miniprogram/pages/index/index.js`

**修改1：引入命令对象**
```javascript
const _ = wx.cloud.database().command;
```

**修改2：修改查询条件**
```javascript
// 修改前
where: { _openid: openid }

// 修改后
where: {
  _openid: openid,
  deleted: _.neq(true)  // 排除 deleted=true 的记录
}
```

### 4.3 修复效果预期

| 对比项 | 修复前 | 修复后 |
|--------|-------|--------|
| **第1页数据库返回** | 10条（含5条已删除） | 10条（全部未删除） |
| **第1页过滤后** | 5条 | 10条 |
| **首屏显示数量** | 5条 ❌ | 10条 ✅ |
| **需要滑动加载？** | 是（为了看到更多） | 否（首屏就足够） |

## 七、总结

### 7.1 问题根源

| 项目 | 说明 |
|------|------|
| **问题类型** | 数据库查询策略问题 |
| **根本原因** | 先查询再过滤导致每页有效数据不足 |
| **影响范围** | 仅影响首页列表的分页显示 |
| **与敏感词的关系** | ❌ 无关联 |

### 7.2 修复方案

| 项目 | 说明 |
|------|------|
| **修复策略** | 将过滤条件前置到数据库查询层 |
| **技术实现** | 使用 `_.neq(true)` 排除已删除记录 |
| **修复复杂度** | 低（仅修改2处） |
| **风险等级** | 低（不影响其他功能） |

### 7.3 经验教训

1. **分页查询时要注意数据有效性**：如果数据存在大量无效记录（如已删除），应该先过滤再分页
2. **日志调试的重要性**：通过详细的日志追踪，才能准确定位问题的真正原因
3. **不要过早下结论**：一开始怀疑 formatLetters 过滤逻辑，但实际证明逻辑本身没问题，问题出在更上游的数据查询阶段
