## 📋 本规范定位

### 协作关系说明

| 文档 | 定位 | 优先级 | 关系 |
|------|------|--------|------|
| **项目开发规范** (`project_rules.md`) | 项目强制规范 | **必须遵守** | 基础规范，所有开发必须遵循，具有最高优先级 |
| **开发原则规范** (`AGENTS.md`) | 技术设计指导原则 | **建议遵循** | 设计参考，提供技术实践指导，与项目规范冲突时以项目规范为准 |

### 使用原则
1. **先遵守强制规范**：所有代码必须首先符合 `project_rules.md` 的硬性要求（如重构禁令、单次变更限制、安全红线等）
2. **后参考设计原则**：在满足强制规范的前提下，可参考本规范优化代码设计、架构和实现质量
3. **冲突解决**：如本规范与 `project_rules.md` 存在冲突，**一律以 `project_rules.md` 为准**
4. **配合使用**：建议开发者同时阅读两份规范，以编写高质量且符合项目要求的代码

## 🎯 核心设计原则

### 第一性原理思考
- **从原始需求出发**：不要假设用户清楚所有细节，深入理解问题本质
- **目标不清晰时停下来讨论**：如果动机和目标不清晰，及时与用户沟通确认
- **主动提出更优方案**：如果用户建议的路径不是最优路径，主动提出更好的方案

### 问题定位原则
- **逻辑/数据同源**：找问题的 root cause 必须使用直接证据，禁止使用间接证据
- **结论必须有支撑**：所有结论都要有直接的代码或数据支撑

### 接口设计原则
- **最低认知负担**：工具和接口的设计目标是"让推理器在最低认知负担下稳定做对下一步动作"
- **简单明确**：优先提供简单、明确的接口，避免复杂的参数设计

## 📝 代码编写原则

### 1. 注释与文档
- 所有函数必须提供完整中文 docstring（参数、返回值、异常）
- 函数内复杂逻辑必须有中文行内注释说明意图
- 类与模块需要有概览性中文 docstring
- **注释要说明"为什么这么做"，而不是"做了什么"**

### 2. 代码结构
- 原则上不使用嵌套函数；仅在必须依赖闭包时例外
- 辅助函数应优先定义为模块级私有函数（`_helper_function`）
- 原则上不使用嵌套类；仅在必须依赖闭包语义时例外
- 保持函数扁平化，优先可测试性与可维护性
- 模块间依赖最小化，优先接口/协议而非具体实现
- 数据处理、存储、工具调用职责明确分离
- 重复逻辑必须抽取为公共函数/类
- 禁止魔法数字/魔法字符串（确有助于可读性的场景除外）

### 3. 实现策略
- 设计前先选择最佳实践方案，直接按最佳实践实现
- **不为历史包袱妥协**（注：但需遵循 project_rules.md 的重构禁令，不得一次性大规模重构）
- 编写处理规则时禁止硬编码规则，要写自适应规则

### 4. 兼容性策略
- 默认按"全新设计"处理，不要求向后兼容旧实现（注：但需遵循 project_rules.md 的保护现有原型原则）
- 可使用现代 JavaScript/TypeScript 特性
- **无需维护历史版本包袱；必要时可重写并删除旧代码**（注：但需遵循 project_rules.md 的重构规范，逐步优化而非一次性重构）

### 5. 测试策略
- 每完成一处代码修改，应同步编写/更新测试并优先验证通过
- 测试跟着实现边界迁移，而不是用生产代码去兼容旧测试
- 单文件测试覆盖率目标为 ≥80%
- 核心功能必须有单元测试

### 6. 文档同步原则
- 测试通过后，立即同步更新相关 README，保持文档与实现一致
- **以代码为准**，不要让 README 先于代码"设计未来"
- 仅更新 README 中没对齐到代码的部分
- 文档写作优先写"当前怎么用 / 当前怎么工作"，不要写"未来可能会怎样"
- 如果一个概念已经改名，必须全量清理旧名，禁止新旧术语并存

## 🏗️ 模块边界规范

### 前端小程序
- **负责**：用户界面、用户交互、云函数调用
- **不负责**：业务逻辑计算、敏感词检测、AI回复生成
- **依赖约束**：只能通过云函数接口访问后端服务

### 云函数层
- **负责**：业务逻辑处理、AI API 调用、数据库操作
- **不负责**：前端界面渲染
- **边界规则**：
  - 敏感词检测模块统一在云函数层实现
  - 配置文件统一管理
  - 工具类可在云函数间共享，但需复制到各云函数目录

### 数据层
- **负责**：数据存储、数据查询
- **不负责**：业务逻辑处理

2. **冲突解决**：如果方案与本规范或 `project_rules.md` 冲突，**一律以 `project_rules.md` 为准**；不得靠文档例外口径兜底

4. **规范更新**：本规范需要更新时，必须经过用户确认

## 🔄 与 project_rules.md 的协作关系说明

本规范（AGENTS.md）与 `project_rules.md` 是**协作互补**的关系：

| 场景 | 规范使用方式 |
|------|-------------|
| 开始新任务时 | 先阅读 `project_rules.md` 了解强制性规范，再参考本规范进行技术设计 |
| 编写代码时 | 优先满足 `project_rules.md` 的硬性要求（如重构禁令、单次变更限制），再应用本规范的代码设计原则 |
| 规范冲突时 | **一律以 `project_rules.md` 为准**，本规范仅供参考 |
| 提交代码时 | 确保代码同时满足两份规范的要求（在 `project_rules.md` 约束范围内应用本规范） |

**总结**：`project_rules.md` 是**底线和约束**，本规范是**优化和提升**，两者配合使用才能编写出既符合项目要求又高质量的代码。
