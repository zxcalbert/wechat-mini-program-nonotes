# 敏感词警告机制优化方案分析报告

## 一、问题描述

**当前问题**：系统在用户提交内容时会显示黄色警告框，但警告显示时间过短，导致用户无法清晰阅读提示内容就完成了记录保存并离开write界面。

---

## 二、方案对比

### 2.1 方案一：延长警告框显示时间

**方案描述**：
延长警告框显示时间至3-5秒，确保用户有足够时间阅读警告内容。

#### 技术实现难度

| 难度项 | 评估 | 说明 |
|-------|------|------|
| 代码修改量 | 🟢 低 | 只需修改定时器参数 |
| 测试复杂度 | 🟢 低 | 只需验证显示时间 |
| 风险 | 🟢 低 | 不影响核心逻辑 |

**实现方式**：
```javascript
// 当前：2秒
wx.showToast({
  title: '笔记已保存',
  icon: 'success',
  duration: 2000
});

// 方案一：3-5秒
wx.showToast({
  title: '注意：您的内容包含投资相关词汇...',
  icon: 'none',
  duration: 4000
});
```

#### 用户体验影响

| 影响项 | 评估 | 说明 |
|-------|------|------|
| 阅读时间 | 🟢 充足 | 3-5秒足够阅读 |
| 操作流畅性 | 🟡 中 | 等待时间略长 |
| 打扰程度 | 🟡 中 | 较长时间显示 |

#### 误操作风险

| 风险项 | 评估 | 说明 |
|-------|------|------|
| 用户忽略警告 | 🟡 中 | 用户可能仍不阅读 |
| 意外关闭 | 🟢 低 | 警告会自动关闭 |
| 误提交 | 🔴 高 | 用户仍可能误提交 |

#### 开发成本

| 成本项 | 评估 | 说明 |
|-------|------|------|
| 开发时间 | 🟢 低 | 5-10分钟 |
| 测试时间 | 🟢 低 | 10-20分钟 |
| 维护成本 | 🟢 低 | 几乎不需要 |

### 2.2 方案二：弹出确认对话框

**方案描述**：
当检测到敏感词时，弹出确认对话框，明确提示"检测到敏感词，是否需要修改？"，提供"修改"和"继续提交"两个选项，由用户主动选择后续操作。

#### 技术实现难度

| 难度项 | 评估 | 说明 |
|-------|------|------|
| 代码修改量 | 🟡 中 | 需要修改提交逻辑 |
| 测试复杂度 | 🟡 中 | 需要测试两种选择 |
| 风险 | 🟢 低 | 逻辑清晰可控 |

**实现方式**：
```javascript
async submitLetter() {
  // ... 其他验证 ...

  const sensitiveCheck = this.checkSensitiveWords(this.data.content);

  if (!sensitiveCheck.canSubmit) {
    wx.showModal({
      title: '内容包含敏感词',
      content: sensitiveCheck.message,
      showCancel: false,
      confirmText: '我知道了'
    });
    return;
  }

  if (sensitiveCheck.message) {
    wx.showModal({
      title: '温馨提示',
      content: sensitiveCheck.message + '\n\n是否需要修改内容？',
      confirmText: '继续提交',
      cancelText: '修改内容',
      success: (res) =&gt; {
        if (res.confirm) {
          // 用户选择继续提交
          this.doSubmit();
        } else {
          // 用户选择修改，不做任何操作
          return;
        }
      }
    });
  } else {
    // 无敏感词，直接提交
    this.doSubmit();
  }
},

doSubmit() {
  // 原有的提交逻辑
}
```

#### 用户体验影响

| 影响项 | 评估 | 说明 |
|-------|------|------|
| 阅读时间 | 🟢 充足 | 用户必须阅读才能操作 |
| 操作流畅性 | 🟡 中 | 需要额外点击确认 |
| 打扰程度 | 🔴 高 | 强制弹窗，打断用户 |
| 选择权 | 🟢 好 | 用户有选择权 |

#### 误操作风险

| 风险项 | 评估 | 说明 |
|-------|------|------|
| 用户忽略警告 | 🟢 低 | 用户必须处理弹窗 |
| 意外关闭 | 🔴 高 | 弹窗可能被误操作 |
| 误提交 | 🟢 低 | 用户有明确选择 |

#### 开发成本

| 成本项 | 评估 | 说明 |
|-------|------|------|
| 开发时间 | 🟡 中 | 30-60分钟 |
| 测试时间 | 🟡 中 | 30-60分钟 |
| 维护成本 | 🟢 低 | 逻辑稳定 |

## 三、方案对比总结

|-----|---------------------|-------------------|

## 四、推荐方案

### 4.1 推荐：方案二（确认对话框）

**推荐理由**：

1. **用户选择权**：用户有明确的选择，体验更好
2. **误操作风险低**：用户必须处理弹窗，不会误提交
3. **阅读充分**：用户有足够时间阅读警告内容
4. **符合设计原则**：强制确认可避免后悔

**权衡**：
- 打扰程度较高，但这是必要的
- 开发成本中等，但值得投入

### 4.2 实施方案二的建议

**具体实现要点**：

1. **投资建议（中敏感）**：
   - 弹窗提示："检测到投资相关词汇，AI回复不会提供具体投资建议。是否继续提交？"
   - 选项："继续提交"、"修改内容"

2. **高敏感词**：
   - 弹窗提示："您的内容包含敏感词，请修改后再提交。"
   - 选项："我知道了"（只有确认，没有取消）

3. **交互优化**：
   - 弹窗文案清晰明确
   - 按钮文字符合用户预期
   - 弹窗出现时不自动关闭

## 五、风险与应对

### 5.1 方案二风险

| 风险 | 说明 | 应对 |
|-----|------|------|
| 用户觉得麻烦 | 弹窗过多可能影响体验 | 只在必要时弹窗（投资建议和高敏感） |
| 误操作弹窗 | 用户可能误点 | 按钮文字清晰，避免误导 |
| 增加开发时间 | 需要更多时间开发 | 权衡后认为值得 |

---

## 六、结论

**最终建议**：采用方案二（确认对话框）

**理由**：
1. 用户有选择权，体验更好
2. 误操作风险低
3. 用户有足够时间阅读
4. 符合设计原则，避免后悔

**实施建议**：
- 投资建议：提供"继续提交"和"修改内容"选项
- 高敏感词：只提供"我知道了"，阻止提交
- 弹窗文案清晰明确，按钮文字符合用户预期
